# 任务复杂度判断指南 v0.0.1

## 📋 版本说明

- **版本**: v0.0.1 (初始版本)
- **状态**: 基础判断标准，会根据使用反馈持续优化
- **目标**: 为AI提供清晰的复杂度判断依据

## 🎯 判断原则

### 核心维度
1. **技术难度**: 需要的专业知识深度
2. **时间需求**: 预估完成时间
3. **协作需求**: 是否需要多人/多Agent协作
4. **影响范围**: 任务的影响面和重要性

### 判断流程
1. 阅读任务描述，识别关键要素
2. 对照L1/L2/L3标准进行分级
3. 考虑边界情况和特殊场景
4. 给出判断理由和置信度

## 📊 复杂度分级标准

### L1 - 简单任务

#### 特征描述
- **操作类型**: 查询、获取、读取、检查、显示
- **技术要求**: 无需专业知识，基础操作
- **时间范围**: 5-15分钟
- **协作需求**: 无需协作，单人可完成
- **处理方式**: 直接返回执行建议

#### 典型关键词
- 查询、获取、读取、检查、显示、列出
- 状态、信息、数据、记录
- 简单、基础、快速

#### 示例任务
```
✅ L1 示例:
- "查询用户ID为12345的基本信息"
- "检查系统当前运行状态"
- "获取最新的订单列表"
- "显示服务器磁盘使用情况"
- "读取配置文件内容"
```

#### 执行策略
- 直接提供操作步骤或命令
- 给出查询方法和工具建议
- 无需分配给专业Agent

### L2 - 专业任务

#### 特征描述
- **操作类型**: 分析、设计、编写、优化、评估
- **技术要求**: 需要专业知识和经验
- **时间范围**: 30-90分钟
- **协作需求**: 单个专业人员可完成
- **处理方式**: 分配给相应领域的专业Agent

#### 典型关键词
- 分析、设计、编写、优化、评估、规划
- 报告、文档、方案、策略
- 性能、安全、架构、流程

#### 示例任务
```
✅ L2 示例:
- "分析网站性能瓶颈并提出优化建议"
- "编写API接口技术文档"
- "设计用户权限管理方案"
- "评估数据库迁移风险"
- "优化前端页面加载速度"
```

#### 执行策略
- 根据任务类型选择专业Agent
- 提供详细的任务描述和要求
- 设定明确的交付标准

### L3 - 复杂任务

#### 特征描述
- **操作类型**: 系统设计、项目规划、多领域整合
- **技术要求**: 需要多领域专业知识
- **时间范围**: 2-6小时或更长
- **协作需求**: 需要多个专家协作
- **处理方式**: 多Agent协调执行

#### 典型关键词
- 系统、架构、平台、整体、全面
- 项目、规划、战略、框架
- 微服务、分布式、集成、协作

#### 示例任务
```
✅ L3 示例:
- "设计完整的微服务电商平台架构"
- "制定公司数字化转型技术规划"
- "构建大数据分析处理系统"
- "设计跨平台移动应用开发框架"
- "规划企业级DevOps流水线"
```

#### 执行策略
- 任务分解为多个子任务
- 分配给多个专业Agent协作
- 需要task-orchestrator进行协调
- 设定阶段性里程碑

## 🤔 边界情况处理

### 模糊任务判断
当任务描述不够清晰时：
1. **询问澄清**: 要求用户提供更多细节
2. **保守估计**: 倾向于选择更高的复杂度级别
3. **分步处理**: 先处理明确部分，再处理模糊部分

### 跨级别任务
当任务跨越多个复杂度级别时：
1. **按最高级别**: 以最复杂的部分为准
2. **任务拆分**: 将复杂任务拆分为多个简单任务
3. **分阶段执行**: 按复杂度递增的顺序执行

### 特殊场景
- **紧急任务**: 可能需要降低复杂度快速处理
- **学习任务**: 可能需要提高复杂度增加教学内容
- **重复任务**: 可能需要降低复杂度提高效率
